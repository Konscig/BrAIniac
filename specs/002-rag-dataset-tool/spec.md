**Feature Branch**: `002-rag-dataset-tool`
**Created**: 2026-04-30
**Status**: Draft
**Input**: User description: "реализуй тул RAG Dataset на основе того что мы обсудили + ответы на вопросы: сделать URI на файл, форматы txt sql и csv, размер ограничить до 1 МБ"

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Подключение Корпуса Документов К RAG-Пайплайну (Priority: P1)

Автор пайплайна собирает RAG-агента в редакторе графа. Ему нужно указать, какой набор документов агент должен использовать как источник знаний (corpus). Сейчас для этой цели используется сущность «Dataset» — таблица в БД, привязанная к пайплайну. Это пересекается с понятием «эталонный датасет» (golden dataset), на котором ИИ-Судья оценивает пайплайн. Чтобы развести эти два смысла, корпус для RAG переезжает в **отдельный тул каталога — `RAG Dataset`** (далее — тул `rag-dataset`). Сущность `Dataset` остаётся исключительно для эталонных пар «вход → ожидаемый ответ» (см. фичу 001-ai-judge).

Автор пайплайна добавляет в граф узел типа «RAG Dataset», в его конфиге перечисляет один или несколько файлов с документами (`txt`, `sql`, `csv`), которые он предварительно загрузил в управляемое хранилище проекта. На вход пайплайна узел не требует никаких данных; на выходе он отдаёт массив документов в формате, совместимом с существующими RAG-тулами (`Chunker`, `Embedder`, `HybridRetriever`).

**Why this priority**: без чёткого разделения «корпус для RAG» и «эталонный датасет для оценки» пользователь вынужден компромиссом класть документы в `Dataset`, что мешает ИИ-Судье и ломает семантику оценки. Это базовый сценарий, без которого корректная работа двух уже существующих фич (RAG-пайплайны и ИИ-Судья) невозможна.

**Independent Test**: автор создаёт пайплайн, добавляет узел `RAG Dataset` со ссылкой на загруженный `corpus.txt` (≤1МБ), соединяет его с `Chunker` → `Embedder` → `VectorUpsert`. Запускает индексацию пайплайна. На выходе индексации в векторном хранилище присутствуют чанки из корпуса. На существующей сущности `Dataset` параллельно создаётся эталонный датасет `gold.jsonl` для оценки — два потока данных не конфликтуют.

**Acceptance Scenarios**:

1. **Given** пайплайн без подключённого корпуса, **When** автор добавляет узел `RAG Dataset` с одним валидным `txt`-файлом до 1МБ и запускает индексацию, **Then** все строки/абзацы файла доходят до Chunker как один документ.
2. **Given** узел `RAG Dataset` с тремя файлами разных форматов (`txt`, `sql`, `csv`), **When** пайплайн исполняется, **Then** на выходе узла отдаются три объекта документов с корректно прочитанным текстовым содержимым каждого файла; порядок документов соответствует порядку, заданному в конфиге узла.
3. **Given** на пайплайне определён узел `RAG Dataset` (для корпуса) и сущность `Dataset` (для эталона), **When** автор запускает оценку через ИИ-Судью, **Then** Судья использует только `Dataset` (golden), а корпус узла `RAG Dataset` не попадает в эталонные пары.

### User Story 2 — Загрузка Файлов Корпуса Через Управляемое Хранилище (Priority: P1)

Файлы корпуса хранятся в управляемом сервером хранилище (по образцу существующих `workspace://`-URI для `Dataset`-аплоадов). Автор пайплайна загружает файл через UI/API, получает в ответ управляемый URI, после чего ссылается на этот URI в конфиге узла `RAG Dataset`. Это устраняет привязку к локальной файловой системе разработчика и делает корпус переносимым между средами.

**Why this priority**: без управляемой загрузки конфигурация пайплайна привязана к путям конкретной машины, что ломает переносимость и работу из CI/SSH-сессий. Это технически независимая часть фичи: можно реализовать узел и без UI-загрузчика (через прямую правку config_json), но переносимый workflow без неё неполон.

**Independent Test**: автор через UI/API загружает `corpus.txt` (700КБ), получает `workspace://datasets/<token>/corpus.txt`. В конфиге узла `RAG Dataset` указывает этот URI. Пайплайн запускается без ошибок и читает реальное содержимое файла из управляемого хранилища.

**Acceptance Scenarios**:

1. **Given** автор имеет доступ к проекту, **When** автор загружает файл `txt`/`sql`/`csv` размером ≤1МБ, **Then** система возвращает управляемый URI и сохраняет файл с проверенной кодировкой UTF-8 (или совместимой).
2. **Given** автор пробует загрузить файл `>1МБ` или с неподдерживаемым расширением (например `pdf`, `docx`, `bin`), **When** запрос приходит на сервер, **Then** загрузка отклоняется с понятным сообщением об ошибке, и файл не сохраняется.
3. **Given** автор указал в конфиге узла URI, по которому файла нет, **When** пайплайн запускается, **Then** исполнение узла завершается с ошибкой, описывающей конкретный недоступный URI, без падения всего пайплайна.

### User Story 3 — Совместимость С Существующими RAG-Тулами (Priority: P2)

Узел `RAG Dataset` отдаёт документы в формате, совместимом с существующими тулами цепочки RAG (`Chunker`, `Embedder`, `HybridRetriever`, `ContextAssembler`). Это обеспечивает «drop-in»-замену существующего узла `DocumentLoader`, который сейчас читает файлы из сущности `Dataset` или произвольных URI.

**Why this priority**: фича должна интегрироваться без переработки нижестоящих тулов. Без этой совместимости автору пришлось бы писать промежуточные адаптеры, что отменяет выгоду от выноса корпуса в отдельный тул.

**Independent Test**: в существующем демо-пайплайне RAG-агента заменяется узел `DocumentLoader` на `RAG Dataset` с теми же исходными файлами. Все нижестоящие узлы работают без изменений конфигурации, а итоговая выдача `LLMAnswer` совпадает по релевантности с исходной (в пределах допустимого шума эмбеддингов).

**Acceptance Scenarios**:

1. **Given** пайплайн `RAG Dataset → Chunker → Embedder → VectorUpsert`, **When** пайплайн запускается, **Then** Chunker корректно принимает выход узла `RAG Dataset` без ошибок типов или схем.
2. **Given** узел `DocumentLoader` уже используется в существующем пайплайне, **When** разработчик добавляет узел `RAG Dataset` в новый пайплайн, **Then** оба узла продолжают работать в своих пайплайнах независимо; `DocumentLoader` не теряет функциональность.

---

### Edge Cases

- **Пустой список файлов** в конфиге узла → исполнение должно завершаться предсказуемо: либо валидация пайплайна на этапе preflight отклоняет такую конфигурацию, либо узел отдаёт пустой массив документов с понятной диагностикой.
- **Файл недоступен** (удалён, неверный URI) → ошибка с указанием конкретного URI, остальные файлы конфига не обрабатываются (fail-fast), чтобы пайплайн не выдавал частичный/неполный корпус незаметно.
- **Файл изначально загружен в пределах лимита, но впоследствии заменён в хранилище на больший** — на момент чтения должна выполняться повторная проверка размера и отказ при превышении 1МБ.
- **Битая кодировка** (например, бинарный файл с расширением `.txt`) → файл отклоняется с диагностикой о невалидном текстовом содержимом.
- **Большое количество файлов** в одном узле — устанавливается верхний предел количества файлов на узел (см. assumption ниже), при превышении конфигурация отклоняется на этапе валидации.
- **CSV без заголовка** или **SQL-дамп** с непечатными символами в строках — текст читается «как есть» в одну строку документа; смысловая разметка (колонки CSV, statement-границы SQL) не интерпретируется на этом этапе и оставлена ответственным нижестоящим тулам.
- **Дубли URI в одном узле** → дубли игнорируются (загружается только первое вхождение каждого URI), о чём фиксируется предупреждение в трассировке исполнения.

## Requirements *(mandatory)*

### Functional Requirements

#### Регистрация И Каталог

- **FR-001**: Система ДОЛЖНА регистрировать в каталоге тулов новую запись с человекочитаемым именем «RAG Dataset» (уникальный код тула — `rag-dataset`), описанием, перечислением поддерживаемых типов узлов, и схемой ожидаемого `config_json` узла.
- **FR-002**: Система ДОЛЖНА предоставлять как минимум один тип узла, ассоциированный с этим тулом, который автор пайплайна может добавить в граф через UI каталога.

#### Конфиг Узла И Источники Файлов

- **FR-003**: Система ДОЛЖНА позволять автору указать в `config_json` узла упорядоченный список ссылок на файлы (URI), составляющие корпус документов узла.
- **FR-004**: Система ДОЛЖНА принимать только URI, ссылающиеся на управляемое хранилище проекта (по образцу существующего префикса `workspace://`); прямые ссылки на произвольные пути файловой системы или внешние URL запрещены.
- **FR-005**: Система ДОЛЖНА разрешать только три типа содержимого по расширению файла: `txt`, `sql`, `csv`.
- **FR-006**: Система ДОЛЖНА отклонять любой файл, размер которого превышает 1 МБ (1 048 576 байт), как на этапе загрузки в хранилище, так и при чтении файла во время исполнения узла.

#### Загрузка И Хранение

- **FR-007**: Система ДОЛЖНА предоставлять API/UI-механизм загрузки файлов в управляемое хранилище, после успешной загрузки возвращающий стабильный URI, который автор может вставить в конфиг узла.
- **FR-008**: Система ДОЛЖНА сохранять файлы в текстовом виде в кодировке UTF-8 (включая ASCII как подмножество); попытка загрузить бинарный/нечитаемый-как-UTF-8 файл должна отклоняться.
- **FR-009**: Загруженные файлы ДОЛЖНЫ быть доступны для чтения нескольким исполнениям пайплайнов в проекте, без дублирующей загрузки на каждый прогон.

#### Исполнение И Контракт Выхода

- **FR-010**: При исполнении узел `RAG Dataset` ДОЛЖЕН прочитать каждый файл из конфига и сформировать массив документов; формат каждого документа ДОЛЖЕН быть совместим с тем, что ожидают на входе тулы `Chunker`, `Embedder`, `HybridRetriever`, `ContextAssembler` (минимально: уникальный идентификатор документа, исходный URI, текстовое содержимое; см. также FR-011).
- **FR-011**: Контракт выхода ДОЛЖЕН точно соответствовать выходному контракту существующего тула `DocumentLoader`, чтобы узел `RAG Dataset` мог использоваться как drop-in замена.
- **FR-012**: Узел НЕ ДОЛЖЕН требовать никаких данных на входе графа (является «source»-узлом для корпуса).

#### Валидация, Ошибки И Диагностика

- **FR-013**: При обнаружении недоступного, отсутствующего или повреждённого файла исполнение узла ДОЛЖНО завершаться ошибкой с указанием конкретного URI и причины (`not found`, `size exceeded`, `invalid encoding`, `unsupported format`).
- **FR-014**: Препроверка пайплайна (preflight) ДОЛЖНА сигнализировать предупреждениями/ошибками, если в конфиге узла отсутствует список файлов, есть дубли URI, превышен лимит количества файлов, или указан запрещённый формат.
- **FR-015**: Все диагностические сообщения ДОЛЖНЫ быть на русском языке для целевой аудитории учебного проекта BrAIniac, с указанием как машинного кода ошибки, так и человекочитаемого пояснения.

#### Обратная Совместимость

- **FR-016**: Сущность `Dataset` (БД) и тул `DocumentLoader` ДОЛЖНЫ продолжать работать без изменений; новый тул `rag-dataset` параллельно сосуществует, не подменяя существующие сущности.
- **FR-017**: Существующие пайплайны без узла `RAG Dataset` НЕ ДОЛЖНЫ требовать миграции данных или повторного развёртывания каталога ноды.

### Key Entities

- **Tool «RAG Dataset» (`rag-dataset`)**: запись каталога тулов, описывающая возможность подключения корпуса документов к графу; не несёт собственного состояния помимо описания и схемы конфига.
- **NodeType «RAG Dataset»**: тип узла графа, использующий тул `rag-dataset`; задаёт схему `config_json` узла и его выходной контракт.
- **Document File**: текстовый файл (`txt`/`sql`/`csv`) размером ≤1МБ, хранящийся в управляемом хранилище проекта и адресуемый стабильным URI.
- **Per-Node Config**: упорядоченный список URI файлов корпуса в `config_json` конкретного экземпляра узла; никакой текст документов в конфиге не дублируется (источник истины — файлы в хранилище).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Автор пайплайна может собрать рабочий RAG-пайплайн (узел `RAG Dataset` → нижестоящие RAG-тулы) на корпусе из одного-трёх файлов меньше чем за **5 минут** с момента открытия пустого пайплайна, не редактируя БД руками.
- **SC-002**: При загрузке файла, превышающего лимит 1МБ, либо с неподдерживаемым расширением, **100%** попыток отклоняются с понятной диагностикой; ни один такой файл не оказывается в управляемом хранилище.
- **SC-003**: Существующая ИИ-судья на эталонном датасете (`Dataset`) производит оценку пайплайна с узлом `RAG Dataset` без ложных срабатываний на корпус: содержимое корпуса не попадает в эталонные пары и не влияет на отчёт оценки. *Гарантируется на уровне дизайна — судья читает только таблицу `Dataset`, корпус живёт в отдельном URI-префиксе и не пересекается с golden-данными; явная regression-проверка не требуется.*
- **SC-004**: Замена `DocumentLoader` на `RAG Dataset` в существующем демо-пайплайне RAG не приводит к падению ни одного нижестоящего узла; итоговая выдача `LLMAnswer` остаётся релевантной (качественно сопоставима с прежней по тем же тестовым вопросам).
- **SC-005**: Каталог тулов в UI отображает «RAG Dataset» как доступный для добавления в граф сразу после развёртывания фичи, без ручного бутстрапа БД пользователем (только через предусмотренные сидинг-скрипты).

## Assumptions

- Управляемое хранилище уже существует в проекте (используется существующим механизмом `Dataset` upload с префиксом `workspace://`) — фича опирается на этот же механизм для хранения файлов корпуса, не вводя нового хранилища.
- Лимит количества файлов на один узел `RAG Dataset` устанавливается равным **64** (по образцу существующего лимита `MAX_DOCUMENT_LOADER_URIS` в `DocumentLoader`); это ограничение разумно для учебных пайплайнов и не блокирует целевые сценарии.
- Текстовая кодировка файлов — **UTF-8** (или ASCII как подмножество); поддержка иных кодировок (windows-1251, cp866 и т.п.) не входит в первый релиз и может быть добавлена позже без изменения внешнего контракта тула.
- CSV/SQL читаются как «сырой текст» — нижестоящие тулы (Chunker, Embedder) сами решают, как обрабатывать структурное содержимое; узел `RAG Dataset` не парсит CSV-колонки и не валидирует SQL-синтаксис.
- Аутентификация и авторизация запросов на загрузку и чтение файлов реализуются существующими механизмами проекта (см. `requireAuth` middleware и проверки `ownership`); фича не вводит новых ролей или политик доступа.
- Существующий `DocumentLoader` остаётся в каталоге как минимум до окончания миграционного периода — новые пайплайны рекомендуется собирать на `RAG Dataset`, существующие могут продолжать использовать `DocumentLoader`.
- Поведение при гонках записи в управляемом хранилище (одновременная замена файла с тем же URI и чтение из исполнения узла) повторяет существующие гарантии хранилища; новые гарантии не вводятся.
