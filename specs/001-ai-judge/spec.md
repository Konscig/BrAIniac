## Clarifications

### Session 2026-04-23

- Q: Модель исполнения оценки (sync vs async)? → A: Async — `POST /judge/assessments` возвращает 202 + `assessment_id` и поддерживает `x-idempotency-key`; прогресс и результат опрашиваются через `GET /judge/assessments/:id`. Шаблон повторяет существующую пару `POST /pipelines/:id/execute` + `GET /pipelines/:id/executions/:executionId`.
- Q: Политика параллельных прогонов одного пайплайна? → A: In-flight lock по `(pipeline_version_id, dataset_id)`; второй запрос без совпадающего `x-idempotency-key` получает `409 Conflict` со ссылкой на активный `assessment_id`. Повтор с тем же idempotency-key продолжает возвращать тот же `assessment_id` (FR-EXEC-001).
- Q: Восстановление при отказе провайдера посреди прогона? → A: Гибрид — per-item retry с bounded-бюджетом попыток и backoff поверх чекпоинтов. Для каждого элемента `(x_k, y_k)` хранится персистентный прогресс; при крэше процесса повторный `POST` с тем же `x-idempotency-key` продолжает с последнего успешно завершённого элемента. При исчерпании бюджета элемент метится `skipped` с классом отказа, assessment продолжается. Весь `JudgeAssessment` уходит в `failed` только при падении доли успешных элементов ниже конфигурируемого порога.
- Q: Интеграция внешних eval-фреймворков (Ragas/DeepEval/Detoxify/BERTScore)? → A: Python-сайдкар `judge-eval-worker` как отдельный контейнер в `docker-compose`, общение по HTTP-JSON (по образцу существующего `tool-executor`). Reference-реализации Ragas/Detoxify/NLI-моделей живут только в Python-сайдкаре; Node вызывает их через клиент-адаптер без прямой зависимости на Python-библиотеки. Native-расчёт в Node остаётся для метрик, не требующих фреймворков (`f_EM`, `f_F1`, `f_recall@k`, `f_ndcg@k`, `f_ctx_prec`, `f_ctx_rec`, `f_toolsel`, `f_argF1`, `f_tool_ok`, `f_trajIoU`, `f_planEff`, `f_node_cov`, `f_schema`, `f_field`, `f_TED`, `f_loop_term`, `f_loop_budget`, `f_loop_conv`, `f_retry`, `f_consist`, `f_check`).
- Q: UX/API загрузки `GoldAnnotation`? → A: Отдельный ресурс в API: `POST /datasets/:id/gold-annotations` поддерживает batch (массив элементов) и single, `PUT /gold-annotations/:id` выполняет ревизию с автоматическим инкрементом версии без потери связей с ранее выполненными оценками. `GET /datasets/:id/gold-annotations` и `DELETE /gold-annotations/:id` дополняют CRUD. Frontend получает ресурсную модель для управления эталонами без смешивания с `POST /datasets/upload`.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Автоматическая Оценка Пайплайна По Базовому Профилю (Priority: P1)

Студент (или сервис-клиент) обращается к API BrAIniac и запускает оценку
существующего пайплайна на эталонном датасете. Среда сама изучает граф,
определяет подмножество метрик `M'_0` по ролям узлов, выполняет прогоны на
`D`, считает значения метрик `S_j` и формирует итоговый отчёт с оценкой `S`,
заключением по порогам `α` и разложением по осям качества.

**Why this priority**: базовый сценарий MVP. Без него пользователь не получает
инструмента объяснимой оценки, и ценность среды BrAIniac как
практикоориентированного инструмента не реализуется. Остальные пользовательские
истории надстраиваются над этим ядром.

**Independent Test**: на демо-пайплайне RAG-агента (узлы `DocumentLoader`,
`HybridRetriever`, `ContextAssembler`, `AgentCall`, `LLMAnswer`) с прикреплённым
эталонным датасетом API-клиент (например, `curl` из [quickstart.md](quickstart.md))
выполняет `POST /judge/assessments`, получает `202 Accepted` + `assessment_id`,
polling'ом `GET /judge/assessments/:id` дожидается `status = succeeded` и
получает отчёт с числовым `S`, вердиктом «доработка / удовлетворительно /
проход», покрытием обязательных осей и поузловой разбивкой метрик.
В данной спеке проверяется публичный API; UI-надстройка над ним выносится в
отдельную последующую спеку.

**Acceptance Scenarios**:

1. **Given** пайплайн с узлами `{Trigger, ManualInput, AgentCall, LLMAnswer}` и
   эталонный датасет на 10 пар `(x_k, y_k)`, **When** пользователь запускает
   оценку, **Then** среда формирует `M'_0 = {f_success, f_passK, f_toolsel,
   f_argF1, f_trajIoU, f_planEff, f_redund, f_node_cov, f_judge_ref, f_consist,
   f_loop_budget, f_faith, f_corr, f_sim, f_cite, f_fact}` без ручной настройки.
2. **Given** тот же пайплайн и датасет, **When** оценка завершается, **Then**
   отчёт содержит значения `S_j ∈ [0, 1]` для каждой метрики, итоговую оценку
   `S = Σ w_j · S_j`, вердикт по порогам `α` (`<0.6`/`0.6..0.8`/`>0.8`) и
   покрытие обязательных осей (Correctness, Grounding, Tool-Use, Safety).
3. **Given** пайплайн содержит цикл `LoopGate -> AgentCall -> LoopGate` с
   `maxIterations = 3`, **When** судья строит отчёт, **Then** метрики
   `f_loop_term`, `f_loop_budget`, `f_loop_conv`, `f_iter_dispersion` попадают
   в `M'_0` и рассчитываются на всех итерациях цикла.
4. **Given** пайплайн не содержит цикла, **When** судья строит отчёт, **Then**
   циклические метрики редуцируются к константам (`f_loop_term = 1`,
   `f_loop_budget = 1`, `f_iter_dispersion = 1`) и не искажают итоговую `S`.
5. **Given** один и тот же граф и один и тот же датасет, **When** оценка
   запускается повторно, **Then** `M'_0` идентичен между прогонами (rule-based
   baseline детерминирован), а отчёт содержит ссылку на `run_id` и версию
   профиля нормализации.

### User Story 2 — Интерактивный Разбор Оценки Через Чат С Судьёй (Priority: P2)

Преподаватель или студент открывает чат с судьёй и задаёт вопросы вида «почему
`f_faith` низкий?», «какой узел тянет оценку вниз?», «покажи trace вызовов
инструментов для узла `AgentCall`». Судья использует инструменты доступа к
графу, метрикам и логам прогонов и даёт обоснованный ответ со ссылкой на
конкретные артефакты.

**Why this priority**: закрывает образовательную петлю — только объяснение
оценки превращает число `S` в инструмент обучения. Ориентация на учебный
сценарий, заявленная в README BrAIniac, требует живого диалога с судьёй.

**Independent Test**: после завершения прогона пайплайна пользователь открывает
панель чата, спрашивает «какой узел дал самый большой вклад в провал?», и судья
корректно указывает узел с наименьшим вкладом в `S`, ссылаясь на `metric_id`
или `run_task_id`.

**Acceptance Scenarios**:

1. **Given** завершённый run с отчётом оценки, **When** пользователь отправляет
   «объясни, почему `f_faith` = 0.42», **Then** судья вызывает инструмент
   доступа к узлу и к трассе выполнения соответствующего `AgentCall` и отвечает
   со ссылкой на конкретные tool-calls и context snippets, которые не
   поддерживают заявленные claims.
2. **Given** чат с судьёй, **When** пользователь запрашивает историю диалога,
   **Then** возвращается упорядоченный список сообщений `user`/`assistant`/`tool`
   с отметками времени.
3. **Given** запрос требует данных, которых у судьи нет (например, несуществующий
   `run_id`), **When** выполняется ответ, **Then** судья возвращает безопасное
   сообщение без галлюцинации значений метрик.

### User Story 3 — История Оценок И Сравнение Прогонов Пайплайна (Priority: P3)

Студент правит конфигурацию агента (например, увеличивает `maxToolCalls`) и
сравнивает новую оценку с предыдущим завершённым прогоном того же пайплайна.
Среда показывает дельты `S`, дельты по осям качества и дельты по отдельным
`S_j`, чтобы студент видел направление изменений и принимал решения об
архитектуре агента итеративно.

**Why this priority**: итеративная доработка — ожидаемый сценарий использования
BrAIniac после MVP. История оценок нужна только если есть стабильный MVP.

**Independent Test**: пользователь (или API-клиент) запускает оценку дважды
для одного `Pipeline` и открывает сравнение; среда показывает, по каким
метрикам произошёл прирост, а по каким регресс.

**Acceptance Scenarios**:

1. **Given** две завершённые оценки (`status = succeeded`) одного `Pipeline`,
   **When** пользователь открывает сравнение, **Then** выводятся `ΔS`, таблица
   `ΔS_j` и изменение вердикта по порогам `α`.
2. **Given** студент удалил узел `HybridRetriever` и запустил новую оценку,
   **When** оценка завершена, **Then** `M'_0` корректно пересобирается без
   метрик оси Retrieval-Качества, а отчёт фиксирует изменение состава `M'`.

### Edge Cases

- Что происходит, если в пайплайне нет ни одного узла, требующего reference
  (только `Trigger` и `SaveResult`)? Ожидаемое: отчёт помечается как «только
  операционные метрики», `S` не рассчитывается, гейт опирается только на
  операционные ограничения.
- Как система обрабатывает unguarded-цикл в графе? Ожидаемое: оценка не
  запускается, возвращается ссылка на `GRAPH_UNGUARDED_CYCLE` из Preflight и
  отчёт пустой.
- Что делать, если на одном из прогонов агент вернул `status = failed`?
  Ожидаемое: прогон учитывается в `R_fail(a)`, качественные метрики считаются
  только на успешных прогонах, в отчёте явно указана доля неуспешных прогонов.
- Как система ведёт себя, если эталонные ответы `y_k` отсутствуют? Ожидаемое:
  метрики оси Correctness/Grounding, требующие reference, исключаются из `M'_0`
  с пометкой причины; в отчёте это отражено как ограничение покрытия осей.
- Что если LLM-судья сам оценивает ответ своего же семейства моделей?
  Ожидаемое: среда выдаёт предупреждение о возможном self-preference bias и
  рекомендует выбрать судью из иного семейства.
- Что если цикл сходится через `stopCondition` раньше `maxIterations`?
  Ожидаемое: `f_loop_budget = 1` на этом прогоне, `f_loop_conv` считается по
  фактическим итерациям.

## Requirements *(mandatory)*

### Functional Requirements

**Анализ графа и выбор метрик**

- **FR-001**: Система MUST принимать на вход агентный граф (Bounded Directed
  Graph) и эталонный датасет `D = {(x_k, y_k)}_{k=1}^m`.
- **FR-002**: Система MUST автоматически строить rule-based baseline `M'_0` как
  объединение рекомендованных подмножеств метрик по ролям узлов, согласно
  таблице «Узел → Стартовое Подмножество M'» из
  [docs/sdd/12-evaluation-metrics-catalog.md](../../docs/sdd/12-evaluation-metrics-catalog.md).
- **FR-003**: Система MUST проверять покрытие обязательных осей качества
  (Correctness, Grounding при наличии контекстного пути, Tool-Use при наличии
  `AgentCall`, Structure при structured output, Safety) и MUST гарантировать
  хотя бы одну метрику в `M'_0` на каждую обязательную ось.
- **FR-004**: Rule-based baseline MUST быть детерминированным: два пайплайна с
  одинаковой архитектурой графа MUST получать идентичный `M'_0`.
- **FR-005**: Судья MUST поддерживать графы с циклами при наличии loop-policy и
  `maxIterations ≥ 1`; ациклическое ограничение из классической математической
  постановки MUST NOT применяться.

**Расчёт метрик и итоговой оценки**

- **FR-006**: Каждая метрика `f_j` MUST выдавать значение на отрезке `[0, 1]`.
- **FR-007**: Система MUST рассчитывать `S_j = (1/m) · Σ f_j(a(x_k), y_k)` по
  всему эталонному датасету.
- **FR-008**: Итоговая оценка MUST вычисляться как `S = Σ w_j · S_j` при
  `Σ w_j = 1`.
- **FR-009**: Для циклических пайплайнов метрики уровня графа (`f_loop_term`,
  `f_loop_budget`, `f_loop_conv`, `f_iter_dispersion`, `f_retry`) MUST работать
  на уровне внешнего графа; метрики траектории внутри `AgentCall` (`f_trajIoU`,
  `f_trajEM`, `f_planEff`, `f_redund`, `f_toolsel`, `f_argF1`, `f_node_cov`)
  MUST учитывать cycle-collapsed нормализацию траектории.
- **FR-010**: Целевые метрики качества (например, `f_faith`) MUST агрегироваться
  по итерациям цикла усреднением или выбором значения последней итерации.
  Выбранный способ (`mean_over_iterations` / `last_iteration`) MUST сохраняться
  в поле `aggregation_method` записи `MetricScore` и MUST отражаться в
  `summary.metric_scores[].aggregation_method` контракта отчёта
  ([contracts/judge-assessments.md](contracts/judge-assessments.md)).
- **FR-011**: Для ациклических конфигураций циклические метрики MUST
  редуцироваться к константам (`f_loop_term = 1`, `f_loop_budget = 1`,
  `f_iter_dispersion = 1`).
- **FR-012**: Система MUST предоставлять единый контракт метрики через
  абстракцию («метрика как абстрактный класс»), от которой наследуются все
  конкретные реализации; новые метрики MUST подключаться без изменения ядра
  судьи.

**Назначение весов и пороги**

- **FR-013**: Стартовые веса `W_0` MUST назначаться по шаблону архитектурного
  класса агента (RAG / Tool-use / Extractor / Judge), описанному в каталоге
  метрик.
- **FR-014**: При накоплении `N ≥ 50` прогонов система SHOULD пересчитывать
  веса через гибридный подход AHP + CRITIC с параметром `λ`, снижающимся по
  мере роста `N`.
- **FR-015**: Система MUST поддерживать пороги интерпретации `α`: `S < 0.6` —
  «доработка»; `0.6 ≤ S ≤ 0.8` — «удовлетворительно»; `S > 0.8` — «проход».
  Пороги SHOULD быть параметризуемыми на уровне профиля оценки.

**Операционные ограничения и жёсткий гейт**

- **FR-016**: Система MUST логировать операционные метрики `T(a)` (p95
  задержки), `C(a)` (суммарная стоимость по токенам и tool fees), `R_fail(a)`
  (доля неуспешных прогонов с таксономией отказов) отдельно от `S`.
- **FR-017**: Система MUST реализовывать жёсткий гейт
  `pass ⟺ S ≥ α_pass ∧ R_fail(a) ≤ R_fail_max ∧ p95(T(a)) ≤ T_max ∧ C(a) ≤ C_max
  ∧ f_safe ≥ 0.95`.

**Нормализация и воспроизводимость**

- **FR-018**: Неограниченные метрики (latency, cost, tokens, TED) MUST
  нормализоваться inverse min-max с перцентильным клиппингом p5/p95.
- **FR-019**: Параметры нормализации (`x_min`, `x_max`, `p5`, `p95`) MUST
  версионироваться совместно с профилем оценки и MUST фиксироваться в отчёте.
- **FR-020**: Каждый отчёт MUST ссылаться на конкретный `run_id`, набор
  `run_task_id` по узлам, версию датасета и версию профиля `(M', W, α,
  normalization_params)`.

**Модель исполнения оценки**

- **FR-EXEC-001**: Оценка MUST запускаться как асинхронная джоба:
  `POST /judge/assessments` возвращает `202 Accepted` с телом, содержащим
  `assessment_id`, `status ∈ {queued, running, succeeded, failed}`, `created_at`
  и отражение `request`. Повтор с тем же заголовком `x-idempotency-key` MUST
  возвращать тот же `assessment_id` без создания повторной джобы.
- **FR-EXEC-002**: `GET /judge/assessments/:id` MUST возвращать текущее
  состояние джобы со стабильными верхнеуровневыми полями: `assessment_id`,
  `run_id?`, `pipeline_id`, `pipeline_version_id`, `dataset_id`, `status`,
  `progress?` (метрика прогресса: `completed_items / total_items`),
  `created_at`, `updated_at`, `started_at?`, `finished_at?`, `request`,
  `summary?` (финальный отчёт при `status = succeeded`), `error?` (при
  `status = failed`).
- **FR-EXEC-003**: Лайфцикл `JudgeAssessment` MUST быть детерминированным:
  `queued → running → succeeded | failed`. Переходы в обход этой последовательности
  MUST NOT происходить.
- **FR-EXEC-004**: Клиент MUST иметь возможность опросить статус polling'ом
  `GET /judge/assessments/:id` с интервалом, определяемым клиентом. Сервер
  MUST отвечать без удержания соединения дольше стандартного HTTP-таймаута.
- **FR-EXEC-005**: Для каждой пары `(pipeline_version_id, dataset_id)` MUST
  допускаться не более одной in-flight оценки со статусом `queued` или
  `running`. Запрос `POST /judge/assessments` на ту же пару без совпадающего
  `x-idempotency-key` MUST возвращать `409 Conflict` с телом, указывающим
  `active_assessment_id`. In-flight claim MUST создаваться атомарно (по
  образцу snapshot store executor'а) и MUST вытесняться по stale-policy на
  основе `updated_at` и конфигурируемого порога.
- **FR-EXEC-006**: Повтор `POST /judge/assessments` с совпадающим
  `x-idempotency-key` (FR-EXEC-001) MUST возвращать тот же `assessment_id` и
  MUST иметь приоритет над проверкой in-flight lock из FR-EXEC-005.
- **FR-EXEC-007**: Каждый элемент датасета `(x_k, y_k)` MUST обрабатываться с
  ограниченным бюджетом ретраев (`max_attempts_per_item`) и экспоненциальным
  backoff между попытками. При исчерпании бюджета элемент MUST помечаться
  статусом `skipped` с классом отказа из таксономии каталога метрик
  (`timeout` / `tool error` / `parsing violation` / `budget exhaustion` /
  `hallucinated tool` / `infinite loop` / `safety abort`) и участвовать
  только в подсчёте `R_fail(a)`; качественные метрики `S_j` для `skipped`
  элементов MUST не считаться, а их доля MUST учитываться при агрегации.
- **FR-EXEC-008**: Прогресс оценки MUST сохраняться в персистентный
  чекпоинт-стор на уровне элемента: для каждого `(x_k, y_k)` фиксируется
  терминальный статус (`completed` / `skipped`), число использованных
  попыток и, при успехе, ссылки на `run_task_id` и значения рассчитанных
  метрик. Повторный `POST /judge/assessments` с совпадающим
  `x-idempotency-key` MUST возобновлять работу с первого элемента, не
  находящегося в терминальном статусе; ранее завершённые элементы MUST NOT
  переисполняться.
- **FR-EXEC-009**: Весь `JudgeAssessment` MUST переходить в `status = failed`
  только при выполнении одного из условий: (a) доля `skipped` элементов
  превысила конфигурируемый порог (`R_fail_max`, по умолчанию `0.5`,
  задаётся через env `JUDGE_FAIL_RATE_MAX`), либо (b) возникла неустранимая
  ошибка уровня оценки (невалидный `dataset`, нарушение Preflight). Во всех
  остальных случаях assessment завершается со `status = succeeded` с
  фиксацией доли `skipped` в `OperationalMetrics.fail_rate` и таксономии
  отказов.
- **FR-EXEC-010**: Размер датасета, допустимый для одного `JudgeAssessment`,
  MUST быть ограничен параметром `max_items_per_assessment` (env
  `JUDGE_MAX_ITEMS_PER_ASSESSMENT`, по умолчанию `500`). При превышении
  `POST /judge/assessments` MUST возвращать `400 Bad Request` с кодом
  `JUDGE_DATASET_TOO_LARGE`.

**Интерактивный судья и чат**

- **FR-021**: Система MUST предоставлять HTTP-маршруты `POST /judge/chat` для
  отправки сообщения судье и `GET /judge/history` для получения истории
  диалога, функционально совместимые с поведением референсной реализации на
  ветке `judge-agent`.
- **FR-022**: Судья MUST иметь доступ к графу, метрикам и логам прогонов через
  tool-calls: как минимум `getNode`, `getMetrics`, `getLogs` с семантикой,
  описанной в референсной реализации `judge-agent`.
- **FR-023**: При отсутствии данных по запросу (несуществующий `id`, отсутствие
  прогона) судья MUST возвращать объяснимое сообщение без фабрикации числовых
  значений.
- **FR-024**: История диалога MUST сохраняться между сообщениями в рамках
  одного контекста чата и MUST быть привязана к пользователю, а не храниться в
  process-local переменной модуля.

**Архитектурные ограничения реализации**

- **FR-ARCH-001**: HTTP-маршруты (`*.routes.ts`) MUST обращаться к данным
  только через сервисный слой (`services/*.service.ts`) и MUST NOT вызывать
  Prisma / ORM напрямую. Это условие явно требует отступления от паттерна,
  существующего на ветке `judge-agent` (прямой доступ к ORM из `metric.service`
  и обход сервисного слоя в `judge.chat.routes`).
- **FR-ARCH-002**: Состояние диалога судьи MUST храниться персистентно
  (в БД с привязкой к пользователю/проекту) и MUST NOT сохраняться как
  process-local in-memory поле инстанса сервиса; инстанс судьи MUST быть
  безсостоянным относительно истории диалога.
- **FR-ARCH-003**: Регистрация инструментов судьи (tool-calls) и их
  handlers MUST быть отделена от HTTP-слоя и от адаптера LLM-провайдера, чтобы
  замена провайдера или расширение набора инструментов не требовала правок в
  маршрутах.

**Внешний расчёт метрик (Python-сайдкар)**

- **FR-EVAL-001**: Метрики, требующие внешних reference-реализаций
  (`f_faith`, `f_fact`, `f_cite`, `f_corr`, `f_contra`, `f_judge_ref` через
  G-Eval/Prometheus-рубрики, `f_safe` через Detoxify/LlamaGuard, `f_sim`
  через sentence-transformers), MUST рассчитываться в отдельном
  Python-сервисе `judge-eval-worker`, поднимаемом через `docker-compose` по
  образцу существующего `tool-executor`.
- **FR-EVAL-002**: Взаимодействие с `judge-eval-worker` MUST осуществляться
  по HTTP-JSON; Node-бэкенд MUST общаться с ним только через выделенный
  клиент-адаптер (по аналогии с `openrouter.adapter`/`tool-executor`
  клиентами), не тянуть Python-зависимости в Node и не обращаться к Python-
  библиотекам напрямую.
- **FR-EVAL-003**: Реестр метрик (см. FR-012) MUST помечать для каждой
  метрики способ исполнения (`native` / `sidecar`). Native-метрики MUST
  считаться в Node без обращения к сайдкару.
- **FR-EVAL-004**: Если `judge-eval-worker` недоступен (health-check не
  проходит), sidecar-зависимые метрики MUST помечаться в отчёте как
  `unavailable` с классом отказа `sidecar_unreachable`, а покрытие
  обязательных осей MUST пересчитываться на оставшихся метриках. Вся
  оценка MUST NOT падать, пока native-метрики покрывают минимум осей
  согласно FR-003.
- **FR-EVAL-005**: `judge-eval-worker` MUST возвращать значения метрик,
  предварительно нормализованные на `[0, 1]` согласно контракту FR-006;
  любые rescale/clip параметры MUST передаваться клиентом либо храниться в
  версионируемом профиле нормализации (FR-019).

**Схема данных и отчёт**

- **FR-025**: Итоговый отчёт оценки MUST быть сохранён в БД в сущности
  `JudgeAssessment` со ссылками на `Pipeline` и `Dataset`; снапшот выхода
  агента на каждом элементе датасета (`agent_output_json`, `tool_call_trace_json`)
  MUST сохраняться непосредственно в `JudgeAssessmentItem`, а не через
  внешние модели `Run`/`RunTask`. Отчёт MUST содержать: `M'`, `W`, `S_j`,
  `S`, вердикт по `α`, покрытие осей, операционные метрики (`T`, `C`,
  `R_fail`), статус жёсткого гейта, версию профиля нормализации. Введение
  самостоятельных моделей `Run`/`RunTask`/`PipelineVersion` как отдельной
  нормализованной схемы исполнения — pre-requisite feature вне данной спеки.
- **FR-026**: Отчёт MUST позволять поузловую разбивку: для каждого узла графа
  — применённое подмножество метрик, их значения и обоснование включения
  (origin-узел + ось качества).
- **FR-027**: Система MUST поддерживать сравнение двух завершённых
  (`status = succeeded`) `JudgeAssessment` одного `Pipeline` с выдачей
  `ΔS` и `ΔS_j`. До появления отдельной сущности `PipelineVersion` каждый
  ассессмент со `status = succeeded` трактуется как отдельная «версия
  оценки»; семантика сравнения не привязана к модели `PipelineVersion`.

**Политика судьи и анти-bias**

- **FR-028**: Среда SHOULD предупреждать пользователя, если судейская модель
  принадлежит тому же семейству, что и оцениваемая модель (self-preference).
- **FR-029**: Для pairwise-сравнений конфигураций (если применяется) система
  MUST рандомизировать порядок и выполнять двойной прогон со сменой позиций
  (митигация position bias).

**Политика данных при неполных эталонах**

- **FR-030**: Если для элемента датасета отсутствует требуемая reference-разметка
  (нет `GoldAnnotation` нужного `annotation_type` либо метрика помечена в
  `MetricDefinition.requires_reference = true` и разметки нет), система MUST
  исключать такие метрики из агрегации `S_j` для этого элемента, MUST
  фиксировать причину в отчёте и MUST NOT завершать оценку ошибкой. Список
  reference-зависимых метрик определяется значением `MetricDefinition.requires_reference`
  и является авторитетным источником.
- **FR-031**: В MVP состав метрик, получающих реализацию в коде, MUST
  покрывать все обязательные оси качества каталога (A..H) с минимум двумя
  метриками на ось там, где каталог это позволяет, и не менее 25 метрик
  суммарно. Обязательный минимум включает:
  - Ось A (Correctness): `f_EM`, `f_F1`, `f_sim`, `f_corr`.
  - Ось B (Grounding): `f_faith`, `f_fact`, `f_cite`, `f_contra`.
  - Ось C (Retrieval): `f_recall@k`, `f_ndcg@k`, `f_ctx_prec`, `f_ctx_rec`.
  - Ось D (Tool-Use и Траектория): `f_toolsel`, `f_argF1`, `f_tool_ok`,
    `f_trajIoU`, `f_planEff`, `f_node_cov`.
  - Ось E (Структура): `f_schema`, `f_field`, `f_TED`.
  - Ось F (Control-Flow): `f_loop_term`, `f_loop_budget`, `f_loop_conv`,
    `f_retry`.
  - Ось G (LLM-as-a-Judge): `f_judge_ref`, `f_check`.
  - Ось H (Safety): `f_safe`, `f_consist`.

  Дополнительные метрики каталога (`f_bert`, `f_selfcheck`, `f_prec@k`,
  `f_mrr`, `f_map`, `f_hit@k`, `f_trajEM`, `f_redund`, `f_success`, `f_passK`,
  `f_type`, `f_branch`, `f_iter_dispersion`, `f_judge_coh`, `f_judge_flu`,
  `f_refuse`, `f_paraph`) MAY быть добавлены за пределами MVP; подключение
  через наследование от абстракции метрики MUST NOT требовать правок ядра
  (см. FR-012).

**Источник эталонных ответов**

- **FR-032**: Reference-данные `y_k` MUST храниться в отдельной сущности
  `GoldAnnotation`, связанной N:1 с `Document` (или с парой `Dataset` + ключ
  элемента) и связанной N:1 с семантическим типом разметки (answer / claims /
  relevant_docs / tool_trajectory / …). `GoldAnnotation` MUST поддерживать
  версионирование: одну и ту же «золотую» разметку можно обновлять без потери
  связей с ранее выполненными оценками. Метрики, требующие reference, MUST
  обращаться к `GoldAnnotation` через сервисный слой, а не к датасету напрямую.
- **FR-032a**: Частный случай FR-030 применительно к `GoldAnnotation`:
  отсутствие разметки нужного типа для документа подпадает под общее
  правило FR-030 «нет reference-данных → метрика исключается из агрегации
  `S_j`» и MUST обрабатываться той же семантикой, включая запись причины
  в отчёт. Отдельной ветки обработки не существует.
- **FR-032b**: Управление `GoldAnnotation` MUST реализовываться как
  самостоятельный ресурс API, отдельный от `POST /datasets/upload`:
  - `POST /datasets/:id/gold-annotations` — создание одной или нескольких
    разметок; тело запроса MUST принимать либо один объект, либо массив
    объектов с полями `document_id` (или `dataset_item_key`),
    `annotation_type` (`answer` / `claims` / `relevant_docs` /
    `tool_trajectory` / иной), `payload`.
  - `GET /datasets/:id/gold-annotations` — перечисление разметок датасета
    с фильтрацией по `annotation_type` и `document_id`.
  - `PUT /gold-annotations/:id` — ревизия существующей разметки; сервер
    MUST автоматически инкрементировать `version` и MUST сохранять
    предыдущие версии так, чтобы ранее выполненные `JudgeAssessment`
    продолжали ссылаться на зафиксированную на момент оценки версию
    разметки.
  - `DELETE /gold-annotations/:id` — soft-delete с сохранением ссылок
    ранее выполненных оценок.
  - Все эти маршруты MUST подчиняться FR-ARCH-001 (обращение к данным
    только через сервисный слой).

### Key Entities *(include if feature involves data)*

- **JudgeAssessment** — итоговый отчёт оценки одного `Run`. Атрибуты: ссылка
  на `Run`/`PipelineVersion`/`Dataset`, состав `M'`, набор `W`, итоговое
  значение `S`, вердикт (`improvement` / `satisfactory` / `pass`), статус
  жёсткого гейта, версия профиля нормализации, `status ∈ {queued, running,
  succeeded, failed}`, `idempotency_key?`, `progress` (`completed_items` /
  `total_items`), `created_at`, `updated_at`, `started_at?`, `finished_at?`,
  `error?`.
- **MetricDefinition** — определение метрики `f_j` из глобального пула `M`.
  Атрибуты: стабильный код (`f_EM`, `f_faith`, …), ось качества, требует ли
  reference, диапазон выхода (`[0, 1]`), способ нормализации, источник
  (native / ragas / deepeval / …).
- **MetricScore** — конкретное значение `S_j` для узла графа в рамках одного
  `JudgeAssessment`. Атрибуты: ссылка на `JudgeAssessment`, ссылка на `Node`,
  ссылка на `MetricDefinition`, значение в `[0, 1]`, обоснование включения
  (origin-узел + ось).
- **WeightProfile** — набор весов `W`, используемый при расчёте `S`. Атрибуты:
  архитектурный класс агента (RAG / Tool-use / Extractor / Judge), значения
  `w_j` по метрикам, метод назначения (AHP / CRITIC / гибрид), значение `λ`,
  CR для AHP.
- **OperationalMetrics** — операционный слой оценки. Атрибуты: ссылка на
  `JudgeAssessment`, `p95_latency_ms`, `total_cost_usd`, `fail_rate`,
  таксономия отказов по классам (timeout, tool error, parsing violation,
  budget exhaustion, hallucinated tool, infinite loop, safety abort).
- **AxisCoverage** — покрытие обязательных осей качества. Атрибуты: ссылка на
  `JudgeAssessment`, список осей с признаком «покрыта / не покрыта» и
  количеством метрик на оси.
- **NormalizationProfile** — версионируемые параметры нормализации. Атрибуты:
  версия профиля, дата калибровки, параметры `x_min`/`x_max`/`p5`/`p95` по
  метрикам.
- **JudgeConversation** — диалог пользователя с судьёй. Атрибуты: ссылка на
  `User`, опционально на `JudgeAssessment`, список сообщений
  `user`/`assistant`/`tool` с timestamp. Хранение персистентное (см.
  FR-ARCH-002).
- **GoldAnnotation** — эталонная разметка элемента датасета. Атрибуты: ссылка
  на `Document` (или `Dataset` + ключ элемента), тип разметки (`answer` /
  `claims` / `relevant_docs` / `tool_trajectory` / иной), payload по типу,
  версия разметки, автор, createdAt.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь может получить итоговую оценку пайплайна с полным
  отчётом менее чем за 5 минут с момента нажатия «Оценить» на демо-датасете
  из 10 пар при стандартных сетевых задержках провайдера LLM.
- **SC-002**: На идентичном пайплайне и идентичном датасете последовательные
  запуски оценки дают идентичный `M'_0` и расхождение `|ΔS|` в пределах 0.02
  (воспроизводимость rule-based части и устойчивость стохастических судейских
  метрик).
- **SC-003**: Судья в чате корректно указывает узел с минимальным вкладом в
  `S` не менее чем на 90% тестовых сценариев, где такой узел однозначно
  определим по данным отчёта.
- **SC-004**: Мета-метрика согласованности оценок судьи с human-labelled
  корпусом достигает Kendall `τ ≥ 0.4` на валидационном наборе.
- **SC-005**: Внедрение новой метрики в систему (через наследование от
  абстракции) требует не более одного файла реализации и одной регистрации
  метрики в реестре, без изменений в ядре судьи и в маршрутах API.
- **SC-006**: Циклические пайплайны с валидной loop-policy успешно оцениваются
  на 100% тестовых графов без искажения итоговой `S` относительно ациклической
  редукции метрик.
- **SC-007**: Пользователь может сравнить оценки двух версий одного пайплайна
  и увидеть дельты по метрикам за один клик без ручной выгрузки данных.
- **SC-008**: 100% HTTP-маршрутов нового модуля судьи проходят статический
  аудит «route → service → ORM» без прямых вызовов Prisma из маршрутов (см.
  FR-ARCH-001).

## Assumptions

- Эталонный датасет `D` предоставляется пользователем заранее и привязан к
  пайплайну через существующую модель `Dataset`/`Document`.
- Пайплайн уже прошёл Preflight (принцип III конституции): граф валиден,
  циклы guarded, `maxIterations` задан.
- Judge-модель и оцениваемые модели исполняются на внешнем LLM-провайдере;
  API-ключи и лимиты — ответственность инфраструктуры, не MVP.
- Прогоны агента пишутся в `Run`/`RunTask` по существующей схеме, и отчёт
  судьи строится поверх этих записей.
- В рамках MVP реализуется только базовый rule-based пайплайн выбора `M'_0` и
  архитектурные AHP-шаблоны `W_0`; полноценная AHP-анкета и CRITIC-пересчёт —
  follow-up после накопления `N ≥ 50` прогонов.
- Чат с судьёй работает в рамках одного авторизованного пользователя, история
  диалога приватна в пределах проекта.
- Взаимодействие с судьёй и инструменты доступа реализуются на существующем
  backend-стеке (TypeScript/Express/Prisma) и повторяют внешнее поведение
  маршрутов из ветки `judge-agent`, но без архитектурных заимствований:
  - референс `judge-agent` не соблюдает слоистость (прямой доступ к Prisma
    из `metric.service` и из routes);
  - референс `judge-agent` хранит `JudgeAgent.history` как process-local поле
    singleton-инстанса маршрута, что недопустимо для многопользовательской
    среды;
  - референс `judge-agent` смешивает регистрацию tool-схем с конкретным SDK
    LLM-провайдера (`@mistralai/mistralai`) — в реализации это разводится.
- Судейский LLM-провайдер MUST быть спрятан за абстракцией `JudgeProvider`
  (интерфейс + конкретные адаптеры). В MVP адаптеры реализуются как минимум
  для двух провайдеров — Mistral (`ministral-3b-2410` из `@mistralai/mistralai`,
  как в референсе `judge-agent`) и OpenRouter (переиспользование существующего
  runtime-адаптера). Активный провайдер выбирается через конфигурацию
  (env/config), без изменений в маршрутах и сервисе судьи. Это реализует
  политику анти-bias каталога метрик (раздел «Политика Судьи (Anti-Bias)»):
  судейский провайдер SHOULD быть настроен на семейство моделей, отличное от
  оцениваемого.
- Исполнение метрик, опирающихся на внешние фреймворки (Ragas, DeepEval,
  BERTScore), будет обёрнуто поверх абстракции метрики; выбор и подключение
  конкретных фреймворков остаётся вне этой спецификации.
