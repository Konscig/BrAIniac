# Постановка задачи оценки конфигурации агента

Нотация и основная модель соответствуют главе 2 ВКР. Разделы «Декомпозиция оценки по узлам» и «Автоматизированная оценка на базе LLM-as-a-Judge» — расширения уровня реализации; они не меняют формулу итоговой оценки S = Σw_j·S_j и служат для интерпретации и автоматизации.

---

## Дано

Спроектированный агентный пайплайн (результат задачи построения):

**a = ⟨V_a, E_a, C_a⟩**

Постановка задачи построения — см. [math_formulation_draft.md](math_formulation_draft.md).

Эталонный набор данных:

**D = {(x_k, y_k)}_{k=1..m},  m ∈ N**

где x_k — входной вектор, y_k — ожидаемый результат.

Множество метрик качества:

**M = {f_1, f_2, ..., f_t},  t ∈ N**

Для агента a существует свой набор применимых метрик

**M' ⊆ M,  M' = {f_1, f_2, ..., f_p},  p ∈ N**

основанный на конфигурации a (составе V_a и ролях его узлов), и соответствующее ему множество весов

**W = {w_1, w_2, ..., w_p},  w_j ≥ 0**

таких, что

**Σ_{j=1..p} w_j = 1.**    (1)

Для каждой метрики f_j ∈ M' вычисляется среднее значение на датасете D:

**S_j = (1/m) · Σ_{k=1..m} f_j(a(x_k), y_k).**    (2)

Итоговая оценка агента — взвешенная сумма:

**S = Σ_{j=1..p} w_j · S_j.**    (3)

Для оценки допустимости качества агента вводится пороговое значение α ∈ [0, 1], определяющее минимально допустимый уровень качества.

## Эксплуатационные ограничения

При оценке учитываются эксплуатационные характеристики:

- **T(a)** — среднее время выполнения пайплайна;
- **C(a)** — средняя стоимость пайплайна;
- **R_fail(a)** — доля неуспешных запусков.

Пороги: **T_max, C_max, R_fail_max**.

Пайплайн признаётся неудовлетворительным и нуждается в доработке, если:

**T(a) > T_max  ∨  C(a) > C_max  ∨  R_fail(a) > R_fail_max**

независимо от значения S.

## Интерпретация итоговой оценки

Для содержательной интерпретации S вводятся пороговые значения:

- **S < 0.6** — агентный пайплайн нуждается в доработке;
- **0.6 ≤ S ≤ 0.8** — пайплайн считается удовлетворительным, но рекомендуется доработка;
- **S > 0.8** — пайплайн проходит оценку качества.

## Найти

Конфигурацию агента a, при которой

**S ≥ α**

при одновременном соблюдении эксплуатационных ограничений (T ≤ T_max, C ≤ C_max, R_fail ≤ R_fail_max).

## Декомпозиция оценки по узлам (расширение)

Данное расширение не входит в формулу (3), но служит инструментом интерпретации при S < α. Оно используется в реализации для поиска слабого звена.

Каждая метрика f_j ∈ M' привязана к области действия scope(f_j):

- **global** — метрика пайплайна в целом (например, end-to-end корректность итогового ответа);
- **node(v_i)** — метрика конкретного узла v_i ∈ V_a (например, корректность ответа LLMCall или полнота выборки DatasetInput).

Пусть **M'_i = {f_j ∈ M' | scope(f_j) = node(v_i)}** — подмножество метрик, относящихся к узлу v_i. Оценка узла:

**S(v_i) = (Σ_{f_j ∈ M'_i} w_j · S_j) / (Σ_{f_j ∈ M'_i} w_j)**    если M'_i ≠ ∅.

Слабое звено при S < α определяется по приоритету:

1. Если нарушено какое-либо эксплуатационное ограничение — таковое нарушение является корневой причиной.
2. Иначе: **v_weak = argmin_{v_i ∈ V_a, M'_i ≠ ∅} S(v_i)**.
3. Если все узловые оценки S(v_i) удовлетворительны, но S < α — причина в метриках scope = global (например, согласованность ответа с эталоном y_k на всей выборке).

Эта декомпозиция имеет смысл только при наличии в M' метрик уровня узла; минимальный корректный случай — M' состоит исключительно из global-метрик, и декомпозиция возвращает пустое множество node-оценок.

## Автоматизированная оценка на базе LLM-as-a-Judge

Каждая метрика f_j вычисляется одним из двух способов:

### Автоматические метрики

Вычисляются без участия LLM-судьи по runtime-телеметрии прогона:

- `error_rate(a, x_k)` — 1, если прогон завершился ошибкой, иначе 0;
- `tool_success_rate(a, x_k)` — доля успешных tool-calls в прогоне;
- `latency(a, x_k)` — время прогона;
- `cost(a, x_k)` — стоимость прогона;
- `token_usage(a, x_k)` — число использованных токенов.

T(a), C(a), R_fail(a) получаются из этих метрик усреднением по D.

### Метрики через LLM-судью (J)

Вычисляются вызовом внешней модели-оценщика J:

**f_j(a(x_k), y_k) = J(a(x_k), y_k, promptTemplate_j)**

Судья J приводит выход к числовой оценке в интервале [0, 1] по заданному промпт-шаблону. Возможен ансамбль из нескольких судей J_1, ..., J_s с усреднением их оценок.

Примеры метрик класса J:

- `semantic_match(O, y) ∈ [0, 1]` — семантическое соответствие ответа эталону;
- `format_compliance(O, y) ∈ {0, 1}` — соответствие ожидаемому формату вывода;
- `groundedness(O, evidence) ∈ [0, 1]` — обоснованность ответа приведёнными источниками (для RAG-сценариев);
- `node_answer_quality(O_i, y, v_i) ∈ [0, 1]` — качество ответа конкретного узла (scope = node(v_i)).

Промпт-шаблоны для J версионируются и хранятся рядом с метрикой.

## Соответствие архитектуре проекта

- Прогон агента на D выполняется исполнителем пайплайна (см. `docs/sdd/09-backend-runtime-truth-snapshot.md`).
- Автоматические метрики извлекаются из записей `RunTask` (время, стоимость, статус, tool-calls).
- LLM-судья J реализуется поверх `services/core/openrouter/openrouter.adapter.ts` отдельной моделью, независимой от моделей, используемых в V_a.
- Состав M', веса W и пороги α, T_max, C_max, R_fail_max задаются на уровне проекта и/или пайплайна; конкретный механизм хранения и UI — в задаче реализации судьи.
