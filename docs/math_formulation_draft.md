# Постановка задачи проектирования агента

Нотация соответствует главе 2 ВКР. Отличие от ВКР: допускается управляемый циклический граф (см. `docs/sdd/02-graph-constitution.md`), а не строгий DAG. Текст ВКР будет обновлён отдельно.

---

## Дано

Каталог доступных функциональных узлов:

**V_all = {v_1, v_2, ..., v_n},  n ∈ N**

где:
- V_all — множество доступных узлов (каталог NodeType);
- n — количество узлов;
- v_i — функциональный узел, характеризующийся шаблоном конфигурации C_i.

Шаблон C_i определяет параметры узла (имена, типы, области допустимых значений), роль (source/transform/control/sink), диапазоны кардинальности входов/выходов, совместимые форматы и, при необходимости, loop-policy и runtime-бюджеты.

Эталонный набор данных:

**D = {(x_k, y_k)}_{k=1..m},  m ∈ N**

где:
- m — количество пар «вход–выход»;
- x_k — входной вектор;
- y_k — ожидаемый результат.

Глобальный бюджет исполнения (ограничения прогона):

**B = (K_max, T_max^run, Cost_max^run, Tokens_max^run)**

где:
- K_max — максимальное число шагов исполнения пайплайна;
- T_max^run, Cost_max^run, Tokens_max^run — верхние пределы времени, стоимости и токенов одного прогона.

## Найти

Тройку

**a = ⟨V_a, E_a, C_a⟩**

где:
- **V_a ⊆ V_all** — подмножество выбранных функциональных узлов;
- **E_a ⊆ V_a × V_a** — множество ориентированных рёбер, образующих граф **G_a = (V_a, E_a)**;
- **C_a = {c_i | v_i ∈ V_a, c_i ∈ C_i}** — набор конкретных значений параметров конфигурации для каждого выбранного узла.

## Ограничения

### 1. Целостность графа

G_a — связный направленный граф с единственным финальным узлом v_fin ∈ V_a, таким что каждый узел v_i ∈ V_a достижим от входа и из него достижим v_fin.

В отличие от классического DAG, **циклы в G_a допускаются** при соблюдении loop-policy:

- каждое ребро (v_i, v_j) ∈ E_a, принадлежащее циклу, MUST быть закрыто loop-policy с параметром maxIterations ≥ 1;
- при достижении maxIterations runtime MUST завершить цикл согласно политике onLimit.

### 2. Совместимость интерфейсов

Для всех рёбер графа:

**∀(v_i, v_j) ∈ E_a: Out_i ~ In_j**

выход узла v_i семантически и функционально совместим со входом узла v_j.

### 3. Иерархическая ацикличность

Если узел v_i ∈ V_a является вложенным пайплайном G_sub = (V_sub, E_sub), то ни один узел из V_sub не может иметь связи с узлами материнского графа G_a. Вложенный граф подчиняется тем же ограничениям рекурсивно.

### 4. Запускаемость на D

**∀(x_k, y_k) ∈ D** выполнение агентом a(x_k):

- завершается без ошибок;
- не превышает бюджет B (число шагов ≤ K_max, время ≤ T_max^run, стоимость ≤ Cost_max^run, токены ≤ Tokens_max^run);
- возвращает результат y'_k, соответствующий выходу финального узла v_fin.

## Результат задачи построения

Тройка a = ⟨V_a, E_a, C_a⟩, удовлетворяющая ограничениям 1–4, — **работающий агент**, готовый к передаче в задачу оценки.

---

## Связь с задачей оценки

Задача построения предшествует задаче оценки и является её необходимым условием. Без работающего агента оценка не определена. Постановка задачи оценки — см. [math_evaluation_draft.md](math_evaluation_draft.md).

---

## Соответствие архитектуре проекта

- V_all — каталог NodeType, зафиксированный в `docs/sdd/07-mvp-node-catalog.md`; MVP-состав: Trigger, ManualInput, DatasetInput, PromptBuilder, LLMCall, AgentCall, ToolNode, Parser, Filter, Ranker, SaveResult.
- C_i хранится в `NodeType.config_json`; c_i — в `Node.ui_json` конкретного экземпляра.
- Loop-policy и runtime-бюджеты — см. `docs/sdd/02-graph-constitution.md` и `docs/sdd/03-node-role-profiles.md`.
- Проверка ограничений 1–3 выполняется на этапе preflight (см. `docs/sdd/05-preflight-contract.md`), ограничение 4 — динамически на поднаборе D.

## Открытые вопросы

- [ ] Фиксация значений глобального бюджета B (K_max, T_max^run, Cost_max^run, Tokens_max^run) — на уровне проекта или per-pipeline.
- [ ] Политика onLimit по умолчанию для циклов (`break` | `error` | `soft_fail`).
- [ ] Формальный способ проверки семантической совместимости Out_i ~ In_j поверх базовой типовой совместимости.
